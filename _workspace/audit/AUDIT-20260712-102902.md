## 2. Divergências detectadas

### 2.1. CRÍTICA — dois documentos órfãos, 100% superseded

`tekton-passo1-multitenant.md` e `tekton-multitenant-e-playbook.md` foram criados no **first commit** (2026-07-05) — são rascunhos anteriores à reorganização de 2026-07-06 que introduziu `tekton-multitenant.md` como versão final e polida da mesma arquitetura (Padrão B multi-tenant). Evidências:

- Cobrem exatamente o mesmo escopo (motivação, 3 padrões, CEL, etapas de implementação, playbook de nova app) que `tekton-multitenant.md`, mas com narrativa menos madura e numeração de etapas diferente (`1.1`, `1.2`... vs. seções `8`, `9`...).
- **Nenhum arquivo do repositório referencia esses dois arquivos** (`grep -rn "tekton-passo1-multitenant\|tekton-multitenant-e-playbook" *.md` não retornou nenhuma ocorrência, nem no `README.md`).
- `tekton-multitenant.md` é estritamente mais completo e é o que está linkado em todos os outros documentos.

**Tratamento:** movidos para `_workspace/pending-review/` (não apagados, por precaução — regra inviolável #1 do prompt de consolidação).

### 2.2. Conteúdo legado duplicado dentro de `tekton-lab-setup.md`

O próprio arquivo já sinaliza isso nos comentários HTML das linhas 1025-1027:
```html
<!-- Os prompts Gemini foram movidos para gemini-prompts.md -->
<!-- O troubleshooting foi movido para troubleshooting.md -->
<!-- Conteúdo original preservado abaixo para referência -->
```
Mas o conteúdo **não foi removido** — as seções 13 ("Prompts para gerar diagramas com Gemini (legado)", linhas 1029-1190) e 14 ("Troubleshooting", linhas 1194-1454) continuam duplicadas integralmente em `gemini-prompts.md` e `troubleshooting.md` respectivamente. Comparação linha a linha confirma que o conteúdo é idêntico (mesmos sintomas/causas/soluções, mesmos prompts).

**Tratamento:** seções removidas de `docs/01-infraestrutura-base.md` (o arquivo reestruturado), substituídas por link para os documentos canônicos. Nada foi perdido — o conteúdo permanece, na íntegra, em `gemini-prompts.md`/`docs/06-diagramas-prompts.md` e `troubleshooting.md`/`docs/05-troubleshooting.md`, e no histórico do git.

### 2.3. Comandos/YAML canônicos duplicados em múltiplos arquivos

O conjunto RBAC do EventListener + TriggerBinding + TriggerTemplate (`app-template`) + Trigger (`gitlab-push-trigger`) aparece **idêntico** (ou quase — ver 2.4) em três lugares:

1. `tekton-multitenant.md` §9 (etapa de implementação, versão narrativa/pedagógica)
2. `tekton-ci-playbook.md` §4 Passo 3/6/7 (playbook operacional "montar do zero")
3. `fix-pipelinerun-namespace.sh` (script de correção que reaplica os mesmos recursos)

**Canônico:** a versão de `tekton-ci-playbook.md`/`fix-pipelinerun-namespace.sh` (mais recente, já reflete o Padrão B completo com `filter`+overlay `pipeline-name`).

**Tratamento:** extraídos para `yaml/ci/` (Pipelines, RBAC, Triggers) e `scripts/setup/` (sequência de bootstrap). `docs/04-ci-operacional.md` passa a referenciar os arquivos/scripts em vez de repetir o heredoc completo. `docs/02-arquitetura-multitenant.md` mantém os blocos inline (valor pedagógico — o documento existe para *explicar* o CEL passo a passo) com uma nota apontando para o arquivo YAML canônico.

Também duplicados em 2 lugares (mesmo texto, mesmo bloco):
- Pipelines `java-app-pipeline`/`node-app-pipeline`: em `tekton-lab-setup.md` (versão single-tenant, sem overlay CEL), `tekton-multitenant.md` §11, `tekton-ci-playbook.md` §4 Passo 5.
- Playbook "adicionar nova stack": em `tekton-multitenant.md` §13 e `tekton-ci-playbook.md` §5 — praticamente texto idêntico.
- "Como ver o token do webhook", "reiniciar o EL": repetidos em pelo menos 4 arquivos diferentes como one-liners `kubectl`.

**Tratamento:** comandos operacionais repetíveis (ver token, reiniciar EL, listar runs, rotacionar token, diagnosticar EL) migrados para `scripts/ops/*.sh`; os `.md` passam a referenciar `./scripts/ops/<nome>.sh`.

### 2.4. Convenção de nomes — sem conflito real, mas vale registrar

`tekton-lab-setup.md` usa o `TriggerTemplate` **`java-app-template`** (single-tenant, hardcoded para uma stack). `tekton-multitenant.md`/`tekton-ci-playbook.md`/`fix-pipelinerun-namespace.sh` usam **`app-template`** (genérico, Padrão B). Isso não é uma divergência de erro — é a evolução documentada da arquitetura (single-tenant → multi-tenant) — mas um leitor que pule direto para `tekton-lab-setup.md` sem ler a nota de topo pode aplicar o template errado. `docs/01-infraestrutura-base.md` já tem uma nota no topo apontando para a evolução multi-tenant; reforçada na Fase 2.

### 2.5. Referências quebradas

Nenhuma referência quebrada encontrada nos `.md` atuais — todos os links internos (`[texto](arquivo.md#âncora)`) apontam para arquivos existentes. A reestruturação da Fase 2 exige reescrever **todos** esses links para os novos caminhos (`docs/NN-nome.md`), incluindo os caminhos relativos de imagem (`imagens/*.png` → `../imagens/*.png` a partir de `docs/`).

### 2.6. `helm-backlog.md` tem sua própria seção "ADRs" (ADR-01 a ADR-05)

Não é uma divergência com os ADRs de plataforma pedidos na Fase 4 (`docs/decisions/ADR-001...`), pois:
- Numeração de dois dígitos (`ADR-01`) vs. três dígitos (`ADR-001`) — sem colisão de nome de arquivo.
- Escopo diferente: são decisões *propostas* (não aceitas) sobre a migração para Helm, vivem dentro do próprio documento de backlog, não como arquivos individuais.

**Tratamento:** mantidos como estão dentro de `helm-backlog.md` → `docs/roadmap-helm.md`. Adicionada uma nota em `docs/decisions/README.md` esclarecendo a distinção, para não confundir os dois sistemas de numeração no futuro.

### 2.7. YAML — validação de sintaxe

Ambiente de execução desta consolidação **não tem `kubectl` disponível** (é uma máquina diferente do cluster k3s do laboratório), portanto a validação pedida (`kubectl apply --dry-run=client`) não pôde ser executada contra um API server real. Validação alternativa aplicada: parse com `python3 -c "import yaml; yaml.safe_load_all(...)"` em todos os manifestos extraídos para `yaml/` — confirma sintaxe YAML válida, mas não substitui validação de schema do Kubernetes/Tekton. Recomendação registrada no relatório final.

## 4. Conclusão da auditoria

Nenhum problema de segurança ou de sintaxe grave foi encontrado — a documentação estava, em geral, correta e coerente entre si (o time já vinha ativamente evitando divergência entre docs e scripts). O principal ganho desta consolidação é estrutural: eliminar os 2 documentos órfãos, extrair os comandos repetidos para `scripts/`, e dar um lugar fixo (`docs/decisions/`) para as decisões arquitetônicas que hoje só existem espalhadas em prosa dentro dos `.md`.
